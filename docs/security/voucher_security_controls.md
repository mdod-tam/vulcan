# Voucher Security Controls

This guide explains the controls that keep voucher assignment and redemption bounded in MAT Vulcan. It focuses on the current behavior and the places developers should not bypass.

The voucher flow has three main checkpoints:

1. An application must be eligible before a voucher can be issued.
2. A vendor must be approved before processing redemptions.
3. A voucher must pass identity, amount, product, and state checks before value is redeemed.

---

## 1. Voucher Issuance

Vouchers are created only when voucher functionality is enabled and the application is eligible. The feature flag check lives in assignment paths such as `Application#assign_voucher!`; the application eligibility check is `Application#can_create_voucher?`, owned by `VoucherManagement`.

An application is eligible for a voucher when:

- the application fulfillment type is `voucher`
- the application status is `approved`
- required proofs are approved
- disability certification is approved
- the application does not already have a voucher

Voucher assignment is normally reached through:

- `Application#transition_status!`, which enqueues `IssueInitialVoucherJob` after an approved transition for voucher applications
- `Admin::ApplicationsController#assign_voucher`, which calls `Application#assign_voucher!`

`Application#assign_voucher!` locks the application before checking eligibility and creating the voucher. Keep voucher issuance through this path so the one-voucher-per-application rule and audit event stay together.

---

## 2. Voucher Value And Code

Voucher value is calculated from the constituent's disability flags and policy values. It is set when the voucher is created; staff do not manually type a voucher value during normal assignment.

Voucher codes are generated by the model with `SecureRandom.alphanumeric(12).upcase`, and the database has a unique index on `vouchers.code`.

The voucher model validates:

- code presence and uniqueness
- initial value is present and non-negative
- remaining value is present and non-negative
- remaining value cannot exceed initial value

This means the database and model both matter. Do not create vouchers by raw SQL or direct inserts in app code.

---

## 3. Vendor Eligibility

A vendor can process vouchers only when `Users::Vendor#can_process_vouchers?` is true.

Current production checks are:

- vendor authorization status is approved
- a W9 form is attached
- W9 status is approved

`Vouchers::RedemptionService` checks vendor authorization before calling the voucher model. In test environment the service uses a simplified authorization check, so production behavior should be verified through model/service tests that cover `can_process_vouchers?`.

---

## 4. Identity Verification

Before redemption, the vendor portal verifies the constituent date of birth through `VoucherVerificationService`.

Current behavior:

- the submitted DOB must parse as a date
- it must match the application owner's date of birth
- failed attempts are tracked in the session per voucher
- the maximum attempt count comes from `Policy.get("voucher_verification_max_attempts")`, defaulting to 3
- a successful verification stores the voucher ID in `session[:verified_vouchers]`

`Voucher#can_redeem?` does not check identity. Identity verification is an external precondition enforced by the vendor portal and `Vouchers::RedemptionService`.

---

## 5. Redemption Checks

`Vouchers::RedemptionService` owns the redemption orchestration. It checks:

- voucher functionality is enabled
- vendor is authorized
- voucher status is active
- identity verification exists in the session
- amount is greater than zero
- amount does not exceed remaining value
- at least one product was selected

Then it delegates to `Voucher#redeem!`, which checks voucher state and amount again through `Voucher#can_redeem?`.

The model-level redemption check requires:

- voucher status is active
- voucher is not expired
- amount is not greater than remaining value
- amount is at least `Policy.voucher_minimum_redemption_amount`

The transaction model also validates that redemption amounts do not exceed the voucher's remaining value.

---

## 6. State And Expiration

Voucher states are:

- `active`
- `redeemed`
- `expired`
- `cancelled`

An active voucher becomes redeemed when remaining value reaches zero. It can become expired when `issued_at + Policy.voucher_validity_period` has passed. Admin cancellation goes through `Admin::VouchersController#cancel`, which calls `Voucher#cancel!` and logs `voucher_cancelled`.

`CheckVoucherExpirationJob` runs voucher expiration processing through `Vouchers::ExpirationProcessorService`.

---

## 7. Audit And Notifications

Current voucher audit events include:

| Action | Owner |
| --- | --- |
| `voucher_assigned` | `Application#assign_voucher!` |
| `voucher_redeemed` | `Voucher#redeem!` |
| `voucher_cancelled` | `Admin::VouchersController#cancel` |
| `voucher_updated` | `Admin::VouchersController#update` |
| status-change events such as `status_changed_to_expired` | `Voucher` callbacks |

Voucher mailers send assignment, redemption, and expiration messages. They are direct mailer calls, not `NotificationService` records.

Voucher audit display starts from `Vouchers::VoucherAuditLogBuilder` and application audit views also include voucher events.

---

## 8. What Not To Bypass

Do not bypass these owners in normal app code:

- `Application#assign_voucher!` for voucher creation
- `Vouchers::RedemptionService` for vendor redemption
- `VoucherVerificationService` for DOB verification
- `Voucher#redeem!` for voucher balance changes
- `Admin::VouchersController#cancel` / `Voucher#cancel!` for cancellation

Avoid direct `update_all`, raw SQL, or manual status writes for voucher value/state changes unless the work is an explicit data repair with separate review.

---

## 9. Verification Checklist

When changing voucher behavior, check:

- Can an ineligible application receive a voucher?
- Can an application receive two vouchers?
- Can an unapproved vendor redeem a voucher?
- Can redemption happen without session DOB verification?
- Can amount exceed remaining value?
- Can redemption happen below the policy minimum?
- Are `voucher_assigned`, `voucher_redeemed`, and `voucher_cancelled` still logged from one owner each?
- Do voucher mailers still receive the records they expect?

Primary code paths:

- `app/models/concerns/voucher_management.rb`
- `app/models/voucher.rb`
- `app/models/voucher_transaction.rb`
- `app/models/users/vendor.rb`
- `app/services/voucher_verification_service.rb`
- `app/services/vouchers/redemption_service.rb`
- `app/jobs/issue_initial_voucher_job.rb`
- `app/jobs/check_voucher_expiration_job.rb`
- `app/controllers/vendor_portal/vouchers_controller.rb`
- `app/controllers/admin/vouchers_controller.rb`
